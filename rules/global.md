# 🌍 全局研发规范

本文件约定了项目团队的全局研发规范，所有开发人员及 AI 代理在开发此项目时均须严格遵守。

## 1. Git 提交规范
为了维护清晰的代码演进历史，项目严格遵循 **Conventional Commits** 标准，并结合本团队定制要求：

- **格式要求**：`<type>: <描述信息>`（例如：`feat: 新增 AI 会话消息终止功能`）。
- **语言限制**：提交信息必须使用**简体中文**书写，禁止携带 `Co-Authored-By` 等协同签名。
- **正文描述**：如果单次提交包含多项修改，必须使用**无序列表**（`-`）在正文（Body）部分描述具体变更详情。
- **核心变更类型（Type）**：
  - `feat`: 新增业务功能
  - `fix`: 修复 Bug 或逻辑缺陷
  - `docs`: 文档、注释类更新
  - `style`: 代码格式调整（空格、分号等，不影响运行逻辑）
  - `refactor`: 代码重构（不增加新功能或修复Bug的结构调整）
  - `test`: 新增或修改测试用例
  - `chore`: 构建工具、构建流程或外部辅助依赖包的变动
- **提交控制**：AI 代理代表人类开发者所做的一切 Git Commit，必须事先由人类开发者人工审查并授权同意后，方可运行提交。

## 2. 核心业务术语
- **Assistant-AI**：移动端 AI 助手，负责核心 AI 会话与各类智能卡片组件的渲染展示。
- **Customer-AI**：客户端 AI 应用，用于服务平台或外部客群。
- **Shared Package**：项目公共组件与方法的提供者（存放于 `packages/shared`），对外暴露共享的工具函数（Utils）与类型声明（Types）。
- **OpenCode**：会话流中的代码安全隔离渲染器。它负责执行、高亮或者在安全沙箱中预览流式输出中的代码段。
- **SSE (Server-Sent Events)**：服务器发送事件，本系统在拉取大模型流式输出（流式文本打字机效果）时采用的核心网络长连接技术。

## 3. 安全红线底线

> [!CAUTION]
> 任何触犯安全红线的行为都将导致系统稳定性或数据安全风险，属于高风险操作。

- **零凭据硬编码（Zero Hardcoded Credentials）**：
  - 严禁在任何前端组件、Hooks、工具函数或构建脚本中硬编码 API 密钥（Keys）、第三方服务 Token、或私有 API 的绝对域名。
  - 所有需要配置的环境变量，必须统一配置在各自 App 根目录的环境变量配置文件（如 `.env.dev`, `.env.test` 等）中，且以 `VITE_` 前缀对外暴露。
  - 在代码中统一使用 `import.meta.env.VITE_xxx` 获取环境变量。
- **安全沙箱与 HTML 防注入**：
  - 对于大模型（AI）输出的 Markdown 或富文本内容，在使用第三方解析器（如 `markdown-it`）渲染时，必须开启严格的 HTML 过滤与转义，避免跨站脚本攻击（XSS）。
  - 严禁对未经转义过滤的用户原始输入直接采用 React 的 `dangerouslySetInnerHTML` 或 Vue 的 `v-html` 渲染。
- **敏感信息暴露限制**：
  - 在非生产（Pro）环境下，可以使用移动端调试工具 `vconsole` 输出开发日志，但在生产环境打包构建时，必须通过配置 `vite.config` 去除所有的 `console.log` 语句，严禁生产环境控制台输出敏感数据与 Token。
